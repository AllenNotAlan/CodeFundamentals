## 🎯 Why This Project

Your `Program.cs` has one problem (Product of Array Except Self) implemented twice. That's not enough. Monzo's interview includes a take-home or pair coding round — you need to:
- Solve problems fluently without getting stuck
- Communicate your thought process while coding
- Recognise patterns quickly and apply the right approach
- Handle edge cases without being prompted

The target: **50 problems in 8 weeks** (roughly 1 per day, with rest days).

### Daily Routine (30–45 min)

```
1. Pick a problem from the week's category (5 min)
2. Read and understand — restate in your own words (3 min)
3. Think through approach OUT LOUD (5 min)
4. Code the solution (15–20 min)
5. Test with edge cases (5 min)
6. If stuck > 20 min: read the hint, try again
7. If stuck > 30 min: read the solution, then re-implement from memory tomorrow
```

### Weekly Structure

```
Monday:    New pattern introduction (read docs, study 1 example)
Tuesday:   Easy problem from that pattern
Wednesday: Medium problem from that pattern
Thursday:  Medium problem from that pattern (different variant)
Friday:    Review — redo any problem you struggled with this week
Weekend:   One "mock interview" problem (random pattern, timed 25 min)
```

## 🧠 Pattern Recognition Cheat Sheet

Use this to quickly identify which pattern to apply:

```
┌─────────────────────────────────────────────────────────────┐
│  "I see..."                    →  "I should try..."          │
├─────────────────────────────────────────────────────────────┤
│  "Find pair/target sum"        →  HashMap (store complement) │
│  "Sorted array + find pair"    →  Two Pointers              │
│  "Subarray/substring of size"  →  Sliding Window            │
│  "Matching brackets/nesting"   →  Stack                     │
│  "Sorted + find element"       →  Binary Search             │
│  "Tree traversal"              →  DFS (recursive) or BFS    │
│  "Shortest path / levels"      →  BFS                       │
│  "Connected components"        →  DFS / Union-Find          │
│  "Ordering dependencies"       →  Topological Sort          │
│  "Optimal substructure"        →  Dynamic Programming       │
│  "Count ways / min cost"       →  Dynamic Programming       │
│  "k-th largest/smallest"       →  Heap / QuickSelect        │
│  "Frequency / duplicates"      →  HashMap / HashSet         │
│  "Linked list cycle"           →  Fast/Slow Pointer         │
│  "Merge sorted things"         →  Two Pointers / Merge Sort │
└─────────────────────────────────────────────────────────────┘
```

## 🎤 Mock Interview Practice

### How to Simulate a Real Interview

**Option 1: Self-timed (free)**
1. Pick a random problem from the list (use a random number generator)
2. Set a 25-minute timer
3. Talk out loud as you solve it (even alone — this builds the habit)
4. Record yourself if possible — review for clarity

**Option 2: Peer practice (free)**
- Find a practice partner on https://www.pramp.com/ (free mock interviews)
- Or ask a friend/colleague to give you a problem and time you

**Option 3: Paid mock interviews**
- https://www.interviewing.io/ — anonymous practice with real engineers
- https://www.tryexponent.com/ — structured practice with feedback

### Communication Template (use every time)

```
1. "Let me make sure I understand the problem..."
   (Restate inputs, outputs, constraints)

2. "Let me think about a few approaches..."
   (Brute force first, then optimise)

3. "I think the best approach here is [pattern] because..."
   (State the pattern and WHY)

4. "The time complexity will be O(n) because..."
   (State complexity BEFORE coding)

5. [Code the solution, narrating as you go]

6. "Let me trace through this with the example..."
   (Verify with the given example)

7. "Edge cases I'd want to handle: empty input, single element, duplicates..."
   (Show testing mindset)
```

## 💡 Tips for Monzo's Pair Coding Specifically

Based on their blog and interview process:

1. **They value communication over speed** — a slower solution you can explain beats a fast one you can't
2. **They use real-world problems** — expect transaction deduplication, rate limiting, data transformation (not pure algorithm puzzles)
3. **They care about code quality** — name variables well, extract functions, handle errors
4. **They want to see testing instinct** — mention edge cases even if you don't have time to code them all
5. **Ask clarifying questions** — "Should I handle negative amounts?" "Is the input always sorted?" This shows product thinking

```csharp
// Problem: Two Sum (LeetCode #1)
// Pattern: HashMap
// Time: O(n) | Space: O(n)
// Date solved: 2026-05-10
// Notes: Key insight — store complement in dictionary for O(1) lookup

public class Solution
{
    public int[] TwoSum(int[] nums, int target)
    {
        var seen = new Dictionary<int, int>();
        for (int i = 0; i < nums.Length; i++)
        {
            int complement = target - nums[i];
            if (seen.TryGetValue(complement, out int j))
                return [j, i];
            seen[nums[i]] = i;
        }
        return [];
    }
}
```
